MQLite

Core concepts: Queue · Topic · Subscription · Filter · DLQ

One SQLite file is the entire message broker. The model sits closest to Azure Service Bus, but is simplified from first principles. This page explains how it works, and how it compares to the equivalents on Azure / AWS / GCP.

01The whole picture

MQLite has two delivery targets: send straight to a queue, or publish to a topic and fan out. One sentence to remember — a Topic stores nothing; it is just a rule that copies a message into a number of subscription queues. The thing that stores, is consumed, and dead-letters is always a Queue.

Try it — toggle the message's properties and hit Publish. Watch which subscriptions receive a copy, and how each copy gets its own seq_number while sharing one message_id:

tier=gold region=eu
next publish → message_id "order-1"

02The five concepts

Queue is the only entity that actually stores messages; Topic / Subscription / Filter are the routing layer around it.

Q

Queue

A named container, the only place messages live. State machine: active → locked → deleted on complete; abandon/timeout returns to active; reject or delivery-count overflow goes to DLQ.

The DLQ is not another queue — it is the same queue's state='dead_lettered' sub-state, worked with peek / redrive / purge.

T

Topic

A logical name standing for "the set of subscriptions attached to it".

It is not an explicitly created object — it exists exactly while a subscription references it. A zero-subscription topic does not exist, and a topic holds no messages: no inbox, no buffer, no retention.

S

Subscription

The binding from a Topic to an independent backing queue, with an optional Filter.

Each subscription has its own consumption, own seq, own DLQ, and belongs to one topic. "A topic has many queues" precisely means: each subscription owns a queue, and the topic fans out through them.

ƒ

Filter

A single expr-lang boolean expression on a subscription, evaluated per message at publish. Empty = match everything.

subject_parts[0] == "orders" && properties["tier"] == "gold"

Compiled and type-checked at Subscribe; a bad one is rejected (400).

03Provisioning: what you create, what is created for you

You only ever do two kinds of "create". You never create a topic, and never create a subscription's queue yourself.

EntityHow it comes to existPrimitive
Plain Queuecreated explicitly CreateQueue(name) — must exist before Send, else error (no auto-create)
Topicno create op (implicit) springs into existence with its first subscription, gone when the last is removed — no CreateTopic
Subscription + backing queuecreated together, one call Subscribe(topic, name, filter) provisions the backing queue for you
⚠️
Common misconception: there is no "create topic" and no same-named queue behind a topic. The backing queue is named after the subscription, not the topic. You do not pre-create a queue to attach a subscription — one call does it (guard: if name is already a plain queue or a subscription under another topic → ErrNameConflict). 
  You only ever do two kinds of "create":
    CreateQueue(name)             -> one independent work queue (must exist before Send)
    Subscribe(topic, sub, filter) -> auto-creates backing queue=sub + attaches to topic
                                     (the topic comes into being as a side effect)

  Name resolution (send / receive / ... all address by name):
    name has subscription rows -> topic, fan out  ·  else -> existing queue  ·  else -> error
🧭
Difference from ASB: in ASB both the Topic and the Subscription are created explicitly; MQLite makes "create topic" implicit and "create the subscription's queue" automatic — two fewer steps.

04Two IDs: don't conflate them

When a message fans out to many subscriptions these two IDs behave the opposite way — the single easiest thing to trip on.

So after fan-out the two copies share a message_id but not a seq_number; they are two fully independent rows, each settled with its own seq_number + lock_token, with no effect on the other.

05FAQ

Q: A message fans out to two subscriptions — same ID?
message_id is the same (copied verbatim); seq_number is different (each backing queue assigns its own globally-monotonic id). Independent rows, settled separately.
Q: I add a filtered subscription — do historical messages route to it?
No. Fan-out happens only at publish time, against the subscriptions that exist then. No replay, no offset rewind. A new subscription only receives messages published after it (and matching its filter). (Unlike Kafka; same as ASB / SNS+SQS.)
Q: A topic gets a message with no subscriptions — dropped? buffered? a default inbox?
No inbox; not retained. ① subscriptions exist but none match → silently dropped (not an error); ② no subscription targets the name at all → falls back to "must be an existing queue", delivered if one exists, else an error. To retain a message, have a matching subscription before publishing.

✓ Like Azure Service Bus

peek-lock + DLQ + Topic/Subscription + Filter + SequenceNumber + dedup map almost one-to-one. If you know ASB, the learning curve is near zero.

✗ Unlike Kafka

MQLite is subscription-based, keeps no log, cannot replay. It is a "work queue + topic fan-out", not a "message log / stream". For replay / event sourcing, choose something like Kafka.

🧭
MQLite's own trade-off: the whole broker is a single SQLite file — embeddable in-process, zero-ops, agent-friendly (ships an MCP server). The price is exactly the "no replay / no topic retention" simplifications above — a deliberate first-principles design, not a gap.

07Speedbumps

⚠️
  • Treating a Topic as "an inbox that stores my messages" — it doesn't; subscribe first.
  • Assuming fan-out copies share one seq_number — they don't; settle each separately.
  • Assuming a new subscription back-fills history — it can't.
  • Treating the DLQ as a queue you create separately — it is a state of the same queue.
  • Publishing to a zero-subscription topic name expecting a buffer — it looks for a same-named queue, or errors.

08Subscription filters (expr)

A filter is the per-subscription gate in the topic fan-out above. An empty filter matches every message. The expression is type-checked and compiled once when you subscribe (a bad one is rejected immediately with 400 invalid_argument) and run once per message at publish time.

Because the filter runs at publish it sees the message's own fields and timestamps; evaluation is deterministic and replayable (it never reads a wall clock).

Try it — does this message route?

Toggle the sample message, then click a filter to evaluate it live against the current message (illustrative — the real engine runs expr-lang):

subjectorders.eu.new
properties tier=gold region=eu priority=high
body.amount120 body_text"urgent: ship now"

09Setting a filter

SurfaceHow
CLImqlite subscribe orders orders-gold --expr 'properties["tier"]=="gold"'
Go SDKcli.Subscribe(ctx, "orders", "orders-gold", &mqlite.Filter{Expr: `properties["tier"]=="gold"`})
HTTPPOST /mqlite.v1.AdminService/Subscribe body {"topic":"orders","name":"orders-gold","filter":{"expr":"..."}}

Re-subscribing with the same name and a new expr replaces the filter (recompiled on the next publish). Omit the filter (or use an empty expr) to receive everything.

Time

Both are the message's own timestamps (epoch-derived time.Time, UTC), not a wall-clock read.

VariableTypeValue
enqueued_attimewhen the message was published — and since fan-out runs at publish, this is "now"
visible_attimewhen it becomes deliverable: equal to enqueued_at for an immediate send, the scheduled time for a delayed/Scheduled one (never null)

Compute a delay by subtraction (time - time → duration):

visible_at - enqueued_at > days(1) # only route significantly-delayed messages enqueued_at.Hour() >= 9 && enqueued_at.Hour() <= 21 # business-hours publish window

Derived

Computed from the message; always defined (cannot error on absence).

VariableTypeExample
subject_parts[]string"orders.eu.new" → ["orders","eu","new"] — MQTT-style hierarchy: subject_parts[0] == "orders"
body_sizeintbyte length — body_size < 4096 (route by size, not content)
property_keys[]stringsorted property names — len(property_keys) > 0, "tier" in property_keys

Body content

Route on the payload itself. These are projected only when your filter references them, so filters that don't touch the body pay nothing.

VariableTypeExample
body_textstringthe raw body as text — body_text contains "urgent" (always defined; "" for an empty body)
body_jsonmapthe body decoded as a JSON object — body_json.amount > 100, body_json["tier"] == "gold", "k" in body_json
⚠️
body_json is only decoded when content_type looks like JSON (or is unset); an explicit non-JSON type, an empty body, invalid JSON, or a non-object JSON (array/scalar) all yield an empty object {}. So body_json itself is never null — but reaching into an absent field (body_json.amount with no amount) yields nil, and comparing that (nil > 100) is a runtime error → fail-closed (not routed, logged). Guard with a presence check: "amount" in body_json && body_json.amount > 100

11Filter language

Standard expr operators and builtins are available.

  • comparison == != < <= > >=, boolean && || ! (or and/or/not), membership in
  • strings: startsWith, endsWith, contains, matches (regex)
  • collections: len(), all(), any(), none(), filter(), map(), indexing x[0]

Durations

For the time fields, durations are spelled with type-checked helpers (recommended, unambiguous) or an extended duration():

FormMeaning
seconds(n) minutes(n) hours(n) days(n) weeks(n)a duration; days/weeks are fixed (24h / 7d)
duration("90m") duration("1d12h") duration("2w")string form; Go's units plus d (=24h) and w (=7d)

There is no month/year unit — calendars make them ambiguous as fixed spans. For "older than a month", use days(30).

visible_at - enqueued_at == hours(2) visible_at - enqueued_at > duration("1d")

13Filter safety

Filters are safe to accept from untrusted callers — the env is the only input and there is no IO.

  • expr is memory-safe, side-effect-free, and always-terminating by design (no file or network access, no unbounded loops).
  • A filter must be a boolean (expr.AsBool); a non-bool expression is rejected at Subscribe.
  • Resource bounds: the source length and compiled AST size are capped.
🛡️
Fail-closed: if a filter errors at runtime (e.g. subject_parts[3] on a two-part subject) or panics, the message is not routed to that subscription and the error is logged — never a broker crash, never a silent match.
